Kézako le SyliusResourceBundle ?

Le SyliusResourceBundle vous permet de gérer rapidement et simplement vos ressources (via des CRUD) et de créer une API REST.

Développé dans le cadre du projet open source Sylius (Framework E-commerce) créer par Paweł Jędrzejewski.

Quelques chiffres!

  • SyliusResourceBundle : ~60 contributeurs, ~60 stars
  • Sylius : ~200 contributeurs, ~1700 stars

Pourquoi ?

  • Eviter la duplication de code dans vos contrôleurs
  • Automatiser certaines tâches lors du développement de Sylius

Comment ça fonctionne ?

Symfony et les CRUD!

  • Un modèle est un object représente les données
  • Un contrôleur avec ses actions (index, show, create, update, delete)
  • Un object manager permettra de persister ou supprimer
  • Un repository la permettra recupération des données
  • Un formulaire permettra la saisie des informations

Le Contrôleur

    Création du ResourceController :

    • C'est un contrôleur générique
    • Il est intialisé via une configuration

    Utilisation du EventDispatcher (répartiteur d'évènement) :

    • Il permet de personnaliser les actions
    • Solution plus flexible

Doctrine

    Plusieurs drivers disponibles :

    • Doctrine ORM
    • Doctrine Phpcr-ODM
    • Doctrine Mongodb-ODM

    Les repositories par défaut ont été personnalisés :

    • Pagination avec PagerFanta
    • Filtrage, triage avancé

Attention! Ce n'est pas un admin generator

Il faut créer vos formulaires, vos templates et le routing à la main (pour le moment!)

Un petit exemple ?

Créons un CRUD pour gérer des utilisateurs, par exemple.

(Nous supposons que notre modèle et son mapping est déjà créé)

Configurons nos ressources

        
sylius_resource:
    resources:
        live.user:
            driver: doctrine/orm
            classes:
                model: Live\Entity\User
                controller: Sylius\Bundle\ResourceBundle\Controller\ResourceController
                repository: Sylius\Bundle\ResourceBundle\Doctrine\ORM\EntityRepository
            templates: AppBundle:Backend/User

Le nom de la resource est préfixé par ce qu'on appelle le "nom de l'application", il est utilisé comme namespace.

Mais que se passe t'il ?

        
$ php app/console container:debug | grep user
live.manager.user    alias for "doctrine.orm.default_entity_manager"
live.controller.user container Sylius\Bundle\ResourceBundle\Controller\ResourceController
live.repository.user container Sylius\Bundle\ResourceBundle\Doctrine\ORM\EntityRepository

$ php app/console container:debug --parameters | grep user
sylius.config.classes {"live.user": {...}}

Créons notre formulaire

        
live.form.type.user:
    class: Live\Form\Type\UserType
    arguments: [Live\Entity\User]
    tags:
        - { name: form.type, alias: live_user }
  • Le ResourceController va chercher le form dans la registry
  • AbstractResourceType permet de configurer le data_class et le validation_group

Créons nos templates

Création ou modification d'une resource 

        


    {{ form_widget(form) }}

Page détail d'une resource 

        


    
user.firstname

Listing de resources 

        
{% for user in users %}
    {{ user.fistname }}
{% endfor %}
  • Une variable nommée form (FormView) est assignée au template de création et modification
  • La variable contenant la collection de ressource pour le listing correspond à la clé de la ressource plurialisée

Configurons nos routes

        
live_user_show:
    pattern: /{id}
    methods: [GET]
    defaults:
        _controller: live.controller.user:showAction
live_user_index:
    pattern: /
    methods: [GET]
    defaults:
        _controller: live.controller.user:indexAction
live_user_create:
    pattern: /new
    methods: [GET, POST]
    defaults:
        _controller: live.controller.user:createAction
live_user_update:
    pattern: /{id}/edit
    methods: [GET, POST]
    defaults:
        _controller: live.controller.user:updateAction
live_user_delete:
    pattern: /{id}
    methods: [DELETE]
    defaults:
        _controller: live.controller.user:deleteAction

Les Contrôleurs sont définis en tant que service, pensez à les utiliser dans votre routing pour n'avoir qu'à modifier la configuration en cas de changement.

Routing :

  • La création et la modification renvoie vers la route show
  • La suppression renvoie vers la route index

Notre CRUD est prêt! Simple, non ?

Allons un peu plus loin ?

Utiliser un template différent

        
live_user_create:
    pattern: /new
    methods: [GET, POST]
    defaults:
        _controller: live.controller.user:createAction
        _sylius:
            template: LiveBundle:Backend/User:custom_create.html.twig

Rédiriger l'utilisateur

        
live_user_create:
    pattern: /new
    methods: [GET, POST]
        _controller: live.controller.user:createAction
        _sylius:
            redirect: live_user_index
            # Ou
            redirect:
                route: live_user_index
                parameters: { name: resource.name }

Configurer un formulaire différent

        
live_user_create:
    pattern: /new
    methods: [GET, POST]
    defaults:
        _controller: live.controller.user:createAction
        _sylius:
            form: form_alias
            # Ou
            form: Live\Form\Type\OtherUserType

Récupérer des données dans la BDD

        
live_user_index:
    pattern: /
    methods: [GET]
    defaults:
        _controller: live.controller.user:indexAction
        _sylius:
            repository:
                method: myCustomMethod
                arguments: []

Il est possible de surcharger la méthode du repository pour toutes les actions (sauf create)

Usiner votre resource

        
live_user_create:
    pattern: /new
    methods: [GET, POST]
    defaults:
        _controller: live.controller.create:actionAction
        _sylius:
            factory:
                method: customCreateNew
                arguments: []

Il est possible de surcharger la méthode du repository pour toutes les actions (sauf create)

Lister ses ressources

        
live_user_index:
    pattern: /
    methods: [GET]
    defaults:
        _controller: live.controller.user:indexAction
        _sylius:
            # Filtre
            criteria:
                company: "Clever Age"
            # Trie
            sorting:
                updatedAt: asc|desc
            # Paginate
            paginate: 50

Parameter Parser

        
live_user_index:
    pattern: /
    methods: [GET]
    defaults:
        _controller: live.controller.user:indexAction
        _sylius:
            # Filtre
            filterable: true
            criteria: $my_criteria

            # Trie
            sortable: true
            sorting: $my_sorting

            # Paginate
            paginate: $my_paginate

Expression Language

        
app_user_index:
    path: /customer/orders
    methods: [POST]
    defaults:
        _controller: app.controller.order:indexAction
        _sylius:
            method: findOrderByCustomer
            arguments: ['expr:service("security.context").getToken().getUser()']

Configuration globale

        
sylius_resource:
    settings:
        paginate: true
        default_page_size: 10
        sortable: false
        sorting: []
        filterable: false
        criteria: []

Créer son bundle

Besoin de gérer vos utilisateurs dans plusieurs applications, créer un bundle pour mutualiser votre code !

La configuration sémantique

        
bundle_name:
    driver: doctrine/orm
    templates:
        user: LiveBundle:Backend/User
        group: ...
    validation_groups:
        user: [live]
        group: ...
    classes:
        user:
            model: Live\UserBundle\Model\User
            controller: Sylius\Bundle\ResourceBundle\Controller\ResourceController
            repository: Sylius\Bundle\ResourceBundle\Doctrine\ORM\ModelRepository
            form:
                default: Live\UserBundle\Form\Type\UserType
                choice: Live\UserBundle\Form\Type\UserChoiceType
        group: ...

La classe dExtension

  • Votre extension doit étendre AbstractResourceExtension
  • Charger facilement vos fichiers de définitions de services
  • Traiter la configuration du bundle

Utilisation d'un opérateurs sur les bits (bitwise operators)

Mapping de vos modèles

Déclarer votre classe en tant mapped-superclass

Le mapping est réécris (évènement loadClassMetadata) pour permettre d'étendre facilement vos modèles

"Doctrine Mapping Driver"

Fournir plusieurs implementations Doctrine pour un model 

        
class UserBundle extends AbstractResourceBundle
{
    protected function getModelNamespace()
    {
        return 'Live\Model';
    }

    protected function getDoctrineMappingDirectory()
    {
        return 'model';
    }
}

Resolve Target Entity Listener

Permet de ne pas écrire en dur les relations entre différentes entités 

        
class UserBundle extends AbstractResourceBundle
{
    protected function getModelInterfaces()
    {
        return [
            'Live\Model\UserInterface' => live.model.user.class
            'Live\Model\GroupInterface' => live.model.group.class
        ];
    }
}

Ré-écrit les paramètres targetEntity dans le mapping de votre model

Créer son API REST

Besoin d'exposer vos utilisateurs via une API ?

Comment ça fonctionne ?

  • Le ResourceController étend the FOSRestController
  • La sérialization est faite grâce à :
    • JMSSerializerBundle
    • BazingaHateoasBundle

Expose en plus de l'état, les actions / transitions possibles autour de l'objet au travers de liens hypermédias.

Hypermedia As The Engine Of Application State permet de faire un API de niveau 3 Hypermedia Controls

  • implémentation découplée
  • plus besoin de manipuler les URLs
  • forme d'auto-documentation de l’API

Configuration vos resources

  • Initialiser vos resources (créer un bundle ou application)
  • Configurer le FOSRestBundle (format, message d'erreur...)

Le routage

Utiliser le ApiLoader pour générer vos routes automatiquement

        
live_user:
    resource: live.user
    type: sylius.api
        
    
        
$ php app/console router:debug | grep resource
live_api_user_index    GET        ANY    ANY  /user
live_api_user_show     GET        ANY    ANY  /user/{id}
live_api_user_delete   DELETE     ANY    ANY  /user/{id}
live_api_user_create   POST       ANY    ANY  /user
live_api_user_update   PUT|PATCH  ANY    ANY  /user/{id}

Les routes sont construites à partir des resources configurées.

  • live: nom de l'application
  • user: nom de le ressource

Le Formulaire

  • Génération automatique des formulaires
  • Possibilité de toujours les créer à la main

Sérialier vos données

        
# Configuration du sérialiseur
Live\Entity\User:
    properties:
        firstName:
            expose: true
            type: integer
            groups: [Details]
            since_version: 0.3
        lastName:
            expose: true
            type: integer
            groups: [Default, Details]
            until_version: 0.4

Sérialier vos données

Header parser récupére dans les entêtes HTTP la version et les groupes de validation

        
GET /api/user/57 HTTP/1.1
Host: live.com
Accept: application/json; version=1.0.1; groups=Default,Details

Vous pouvez les configurer directement dans vos routes

        
app_user_index:
    path: /user/
    methods: [POST]
    defaults:
        _controller: app.controller.order:indexAction
        _sylius:
            serialization_groups: 1.0.1
            serialization_version: [Default, Details]
