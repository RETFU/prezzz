STUPID, Kézako ?

Ce sont des mauvaises pratiques.

Instances uniques

  • Un singleton est comparable à une variable globale de l'application
  • Un singleton ne facilite pas la configuration des objets
  • Les singletons introduisent des couplages forts entre les objets

Instances uniques (Singleton)

  • Un singleton est comparable à une variable globale de l'application
  • Un singleton ne facilite pas la configuration des objets
  • Les singletons introduisent des couplages forts entre les objets

Couplage fort (Tight Coupling)

  • Difficulté dans la mise en place des tests (impossible de créer des mocks)
  • Dépendence forte entre composants

Non testabilité (Untestability)

  • Eviter bogues, les régressions et faciliter la refacto
  • Un code fortement couplé sera difficilement testable

Optimisations prématurées (Premature Optimizations)

  • La lecture, la compréhension ou/et la maintenance du code peuvent être plus difficiles
  • De simples applications peuvent se transformer en usine à gaz

Nommage indéchiffrable (Indescriptive Naming)

  • Gain de temps lors de la relecture, la maintenance ou l’évolution
  • L’usage d'abréviations ou d'acronymes est fortement déconseillé car tout le monde ne connaît pas forcément le sens exact de ces derniers

Duplications (Duplication)

  • Une modification du code nécessite de répercuter le changement à plusieurs endroits
  • PHP Copy Paste Detector (PHPC) ou SonarSource automatisent la détection de duplications de code

SOLID, Kézako ?

  • Principe Michael Feathers et Robert C. Martin (aka Uncle Bob)
  • Ce sont des bonnes pratiques !

Principe de responsabilité unique (Single Responsability Principle)

  • Une classe doit remplir un rôle précis
  • La classe est facilement testable
  • Facilite l’évolution des implémentations existantes ou d'en ajouter de nouvelles

Principe ouvert / fermé (Open Close Principle)

  • Le principe ouvert / fermé consiste à rendre les modules ouverts à l'extension et fermés aux modifications

Principe de substitution de Liskov (Formulé par Barbara Liskov et Jeannette Wing)

  • Formulé par Barbara Liskov et Jeannette Wing
  • Si la classe T a une dépendance de type S alors on doit pouvoir remplacer cette dépendance par tous types dérivés de S
  • Conserver les signatures des méthodes d'une classe parente ainsi que la nature des valeurs de retour de ces dernières.
  • Diminution du couplage.

Principe de ségrégation d'interfaces (Interface Segregation Principle)

  • Formulé par Barbara Liskov et Jeannette Wing
  • Si la classe T a une dépendance de type S alors on doit pouvoir remplacer cette dépendance par tous types dérivés de S

Principe d'injection de dépendance (Dependency Injection Principle)

  • Injecter dynamiquement les dépendances d’une classe en s'appuyant sur une description ou de manière programmatique
  • Une nette diminution du couplage
  • Une meilleure encapsulation
